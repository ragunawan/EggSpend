**Note (2026-07-09, loop 11 T10):** B12 fixed via T10 (Balance edits as explicit adjustments): Account edit now calls `AccountBalanceService.applyBalanceEdit(oldBalance:newBalance:to:context:)` instead of directly writing account.balance. The helper creates a "Balance adjustment" Transaction (nil category, income-typed if delta>0 else expense, abs(delta) amount) and applies it, yielding exactly the new balance. Excludes isAdjustment transactions from all income/expense/cash-flow/savings-rate calculations (12 summation sites verified: MonthlyReviewCalculator income/expenses/topCategories/budgetOverruns, Budget.spent, MetricsView totalIncome/totalExpenses/expensesByCategory/cashFlowData, DashboardView monthlyIncome/monthlyExpenses, ForecastEngine.averageDailyNetFlow, SafeSpendCalculator.averageDailyExpenses). CRITICAL nil-category trap: nil-category budgets catch uncategorized spend, so an adjustment expense would count against them and fire BudgetAlertCoordinator alerts on account edits (would be a user-visible surprise for a correction transaction). Deliberately INCLUDE adjustments at NetWorthCalculator (adjustments ARE part of balance history; at() must reverse them to reconstruct historical balances correctly), CashFlowCalendarView (day sums), TransactionsListView.dailyNetTotal (headers must add up to visible rows), CSVImportView (parsed rows).
