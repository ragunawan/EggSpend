# Changelog

## Unreleased

- Added App Store encryption compliance metadata declaring no non-exempt encryption.
- Fixed a bug where net worth in the Metrics timeline and Monthly Review was computed by adding liability balances instead of subtracting them, inflating reported net worth; introduced a shared `NetWorthCalculator` now used consistently by Dashboard, Nest Egg, Metrics, and Monthly Review, and Metrics/Monthly Review now respect the "include in net worth" account setting.
- Added GitHub Actions CI workflow (`.github/workflows/test.yml`) with runtime simulator resolution and artifact capture; tests now run on macOS runners for all PRs and pushes to main and claude/* branches.
- Fixed historical net-worth reconstruction in Metrics timeline and Monthly Review to only include account-linked transactions; account-less (unlinked) transactions are now excluded from retroactive net worth calculations, respecting the "include in net worth" account setting consistently.
- Fixed CSV import amount and transaction-type parsing: `parseAmount` now detects sign indicators (parentheses, leading/trailing minus, U+2212) before stripping whitespace and rejects embedded minuses; CSV type inference now uses expense/income keyword lists with amount-sign fallback for unknown types, applying whole-word token matching to short keywords (pos/dr/cr) to prevent false matches like "pos" in "Deposit".
- Fixed recurring transaction materialization for ended items: items that have reached their end date but still have overdue occurrences between nextDueDate and endDate now materialize those final transactions instead of being silently skipped. Also added an infinite-loop guard that halts materialization if `advanceNextDueDate()` fails to strictly advance the date, preventing hangs when the Calendar API returns nil.
- Fixed amount entry broken in comma-decimal locales (fr_FR, de_DE, etc.) by introducing `AmountParser` utility with position-validated grouping fallback; all form fields (transaction amounts, account balances, budget limits, transaction filters) now safely parse locale-specific decimal separators and thousands groupings instead of relying on `Double(text)` which fails when the device locale uses comma for decimals.
- Improved error handling for data persistence failures in critical paths: CSV imports now surface save errors with user-facing messages and allow retry without double-inserting rows; budget alert state changes now explicitly persist instead of silently discarding; recurring transaction processing returns a Bool indicating success/failure instead of silently logging to console; default category seeding returns a Bool on save (failures logged). Failed import saves perform targeted rollback of only the import's own newly-inserted transactions, avoiding silent discard of unrelated unsaved edits from other screens.
- Added account archival as a safer alternative to deletion: accounts can now be archived via a swipe action with confirmation; archived accounts are excluded from net-worth calculations, cash-flow forecasts, and balance pickers (while preserving the ability to view transaction history for archived accounts); accounts can be unarchived from a dedicated archived section, or permanently deleted via a second confirmation; this prevents accidental data loss and orphaning of transaction history.
- Enhanced CSV import workflow to support account selection and optional automatic balance updates: users can now link imported transactions to an account (optional; defaults to None), and choose whether to apply each transaction's amount to the selected account's balance; the preview step shows the calculated net effect on the account balance with clear increase/decrease indication and currency formatting; failed imports now roll back their balance changes symmetrically to prevent data corruption on retry.
- Added duplicate-transaction detection on CSV import: the import preview now identifies transactions that match existing entries in the ledger (same date + amount + title, normalized for case and whitespace) and skips them during import; duplicate count displayed in the import summary; applies to transactions within the same import file and prevents re-importing the same CSV multiple times.
- Fixed account balance editing to record corrections as explicit "Balance adjustment" transactions: account edits now create a visible transaction in the Transactions ledger showing the balance change, making it easy to audit corrections and trace account history. Balance adjustments are excluded from income/expense metrics, spending categories, cash-flow forecasts, and savings-rate calculations to prevent skewing financial summaries; they are included in net-worth calculations to ensure historical balance reconstruction remains accurate. Budget detail screens now consistently match budget totals by delegating to a shared predicate that excludes adjustment transactions.
- Amounts and currency symbols throughout the app now follow the device region's locale: decimal separators (comma in fr_FR, period in en_US), thousands groupings (space/period/comma depending on locale), and currency symbols (£ for GBP, € for EUR, ¥ for JPY) are automatically derived from the device locale instead of being hard-coded to USD; USD remains the fallback for unsupported locales, ensuring familiar behavior on less-common device settings.
- Added data export engine for backup and portability: transactions, accounts, and transfers can be exported as RFC-4180-compliant CSV files with locale-aware formatting; a comprehensive versioned JSON backup envelope (BackupEnvelope format, schema version 1) captures all seven app models (Transaction, Account, Budget, RecurringTransaction, SavingsGoal, Transfer, TransactionCategory) with full fidelity for round-trip validation and future restore workflows. CSV exports include deterministic POSIX date/number formatting to ensure consistency across locales. JSON backup validates schema-version compatibility on decode (rejecting future schema versions with descriptive errors). Export UI (Settings screen with ShareLink/fileExporter) arrives with the next loop (T13b).
