# EggSpend — Audit & Product Roadmap (July 2026)

Prepared by an automated senior-engineer/product review of the codebase at commit `a5f6a25`.

---

## 1. Executive summary

EggSpend is a well-structured, offline-first SwiftUI/SwiftData personal finance app with an unusually complete feature set for its size (~5,900 LOC of app code, 18 test files): transactions, transfers, accounts, budgets with alerts, recurring transactions with reminders, CSV import, savings goals, debt payoff planning, cash-flow forecasting, "safe to spend," monthly review, and net-worth tracking.

The biggest problems are **correctness inconsistencies in money math**, not missing features:

1. **Net worth is computed four different ways in four places**, and two of them (Metrics timeline, Monthly Review) have a **sign bug that adds liabilities to net worth instead of subtracting them**.
2. **Manual account balances and transaction history are two disconnected sources of truth.** Balance updates are the caller's responsibility (`AccountBalanceService`), CSV-imported transactions never touch balances, and historical net-worth "reconstruction" reverses *all* transactions including ones that never affected any balance — so history charts drift from reality.
3. **Locale/currency handling is US-only** (`Double(amountText)` parsing, 76 hard-coded `.currency(code: "USD")` call sites), which breaks amount entry outright on comma-decimal locales.

Recommended killer-feature direction: double down on the already-started **"forward-looking cash calm" suite** (Safe-to-Spend + Forecast + Subscription/recurring intelligence) — but only after Phase 0 stabilization, because insight features built on inconsistent net-worth math will erode trust.

## 2. Repo architecture overview

**Stack:** Swift 6, SwiftUI, SwiftData, Swift Charts, UserNotifications. CloudKit-backed `ModelContainer` with automatic local-only fallback (`EggSpendApp.swift:17-31`). No third-party dependencies, no backend, no auth (data is device/iCloud-private). Project file is generated by `generate_project.py` and must be kept in sync manually.

**Layout:**

| Path | Purpose |
|---|---|
| `EggSpend/Models/` | 7 `@Model` classes: `Transaction`, `TransactionCategory`, `Account`, `Budget`, `RecurringTransaction`, `SavingsGoal`, `Transfer`. Enums stored as raw `String`/`Int` for CloudKit compatibility. |
| `EggSpend/Persistence/` | `PersistenceController` (category seeding, preview container), `SyncStatus`. |
| `EggSpend/Utilities/` | Business logic: `AccountBalanceService`, `TransferBalanceService`, `SafeSpendCalculator`, `MonthlyReviewCalculator`, `DebtPayoffCalculator`, `CSVParser`, `RecurringProjection`, `NotificationScheduler`, `BudgetAlertCoordinator`, `TransactionFilter`. |
| `EggSpend/Views/` | 13 feature folders; `ForecastEngine.swift` lives in `Views/Forecast/` (documented quirk). |
| `EggSpendTests/` | 18 XCTest classes using in-memory containers. No UI test target, no CI. |
| `scripts/`, `screenshots/`, `docs/` | Screenshot capture, App Store assets, GitHub Pages (privacy/support). |

**State management:** Views use `@Query` and pass arrays into pure static calculators. Mutations happen directly in views; account balance side-effects must be invoked manually at every call site.

**Data-model conventions:**
- `Transaction.amount` is always positive; `signedAmount` gives direction.
- Liability `Account.balance` is stored **negative** (`AddAccountView.swift:110`).
- `Budget.category == nil` means "uncategorized spend," not "all spend."
- Recurring transactions materialize into real `Transaction` rows at launch (`processRecurringTransactions`, `RecurringTransaction.swift:127`).

**Run/test:** `xcodebuild build|test -project EggSpend.xcodeproj -scheme EggSpend -destination 'platform=iOS Simulator,name=iPhone 17'`. Launch args `--preview-data`, `--tab <n>`.

**Setup gaps:** no CI workflow; no schema-migration story (SwiftData lightweight migration only — fine today, but any renamed/retyped field will need a `SchemaMigrationPlan`); `generate_project.py` sync is manual and error-prone; no CONTRIBUTING or architecture doc beyond CLAUDE.md.

## 5. Security, privacy, and financial-data risks

The posture is fundamentally good: local + iCloud-private storage, no third-party services, no analytics, `PrivacyInfo.xcprivacy` and a published privacy policy in `docs/`. Remaining gaps:

- **No app lock.** Anyone with the unlocked phone sees all finances. Add Face ID/Touch ID (`LAContext`) app lock with a toggle — standard for finance apps and a trust signal. (M)
- **No file protection assertion.** Rely on default iOS data protection; consider `NSFileProtectionComplete` for the store. (S)
- **Silent data-loss paths:** `try? context.save()` throughout (`PersistenceController`, `CSVImportView.swift:403`) and `print()`-only error handling (`RecurringTransaction.swift:180`) mean a failed save is invisible. Surface save failures to the user at least for imports and deletes. (S)
- **CloudKit fallback silently forks data.** If CloudKit init fails once, the app runs local-only (`EggSpendApp.swift:24-31`); `SyncStatus` exists but verify it's surfaced prominently — users may believe they're backed up when they aren't. Also `try!` on the fallback container will crash if both containers fail. (S)
- **No export/backup.** Users can import CSV but never get their data out — a data-portability and trust issue (and an App Store review talking point). (M)
- **Destructive actions without confirmation** (#11 above) are effectively data-loss bugs.
- No secrets in repo (checked). No network calls at all today — keep it that way for any AI features unless clearly consented (see §6).

## 6. Recommended new features

**Core tracking:** account-aware CSV import (+dedupe) · quick-add widget/shortcut · transaction splitting · attach receipts (photos) · multi-currency (post #6).

**Budgeting:** rollover budgets · overall monthly budget (all categories) · budget suggestions from 3-month category averages ("You spend ~$430/mo on food — set $450?").

**Net worth:** `BalanceSnapshot` model (periodic snapshots instead of reconstruction — fixes #3 structurally) · asset classes (property, vehicle) · net-worth milestones.

**Categorization:** rule-based auto-categorization (merchant substring → category, learned from user's manual edits) — biggest daily-friction reducer; works offline, no AI needed.

**Recurring/subscriptions:** subscription detection from imported history (same title/amount cadence) · "subscription audit" screen with annualized cost and cancel reminders · price-change detection ("Netflix went from 15.49 → 17.99").

**Insights/charts:** "What changed this month?" summary (category deltas vs. 3-month average) · anomaly flags (unusually large transaction for its category) · weekly digest notification.

**AI-assisted (optional, later):** plain-English monthly narrative generated on-device from `MonthlyReviewData`; if a model API is ever used, make it opt-in, send only aggregates (never raw transactions), and say so in the privacy policy. Template-based text gets 80% of the value with zero privacy cost — start there.

**Notifications:** already strong (bill reminders, budget alerts). Add safe-to-spend morning summary (opt-in) and goal-milestone celebrations.

**Import/export:** CSV export (transactions, accounts) · full JSON backup/restore.

**Security/privacy:** Face ID lock · export · visible sync status.

**Onboarding/UX polish:** first-run flow (create accounts → optional import → pick budgets) · empty-state CTAs on Metrics/Forecast · haptics on save · iPad layout audit.

## 7. Killer feature recommendation

Top 3 directions, ranked:

1. **"Cash Calm" — Safe-to-Spend + Forecast + Subscription intelligence as the hero loop.** EggSpend already has SafeSpendCalculator, ForecastEngine, and recurring infrastructure — 70% built. Differentiator vs. Mint-alikes: answer "can I spend $80 tonight?" instantly and trustworthily, offline, private. Requires Phase 0/1 correctness first. **Build this first.**
2. **Subscription leakage detection.** Narrow, marketable, demo-able ("EggSpend found $63/mo you forgot about"). Builds directly on recurring detection and feeds direction 1. Second.
3. **Plain-English "What changed this month?"** Retention driver (a reason to open the app monthly), cheap to build from `MonthlyReviewCalculator` deltas, and an on-ramp to AI later. Third.

Skip for now: goal-based planners beyond current goals, AI chat coaches (privacy cost, high effort), bank aggregation (Plaid — changes the product's privacy story entirely; **needs your decision** before ever pursuing).

## 11. Product decisions (resolved 2026-07-08)

| Question | Decision | Effect on plan |
|---|---|---|
| Bank aggregation (Plaid) | **Long-term goal**, not on the current roadmap. | Keep the offline-first privacy story intact through Phase 3. Design the Phase 1 import/dedupe pipeline so an aggregation source can later feed the same `ParsedTransactionResult` path. No Plaid work now. |
