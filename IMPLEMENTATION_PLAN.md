- **Verified against actual current code this loop (drift check against the audit's line numbers — confirmed NOT stale):**
  - `EggSpend/Views/Transactions/TransactionsListView.swift:85-97` — the `grouped` computed property is unchanged since the original audit and still exactly matches the cited line range:
    ```swift
    private var grouped: [(String, [LedgerRow])] {
        let formatter = DateFormatter.sectionHeader
        var dict: [String: [LedgerRow]] = [:]
        for row in rows {
            let key = formatter.string(from: row.date)
            dict[key, default: []].append(row)
        }
        return dict.sorted { lhs, rhs in
            let d1 = rows.first { formatter.string(from: $0.date) == lhs.key }?.date ?? .distantPast
            let d2 = rows.first { formatter.string(from: $0.date) == rhs.key }?.date ?? .distantPast
            return d1 > d2
        }
    }
    ```
    The O(n²) claim is confirmed live: `rows` is itself a computed property (filter + sort over `filteredTransactions`/`filteredTransfers`/`upcomingRecurring`, i.e. re-derived on every access, not cached), and the sort comparator inside `grouped` calls `rows.first { ... }` — an O(n) linear scan — **twice per comparison**, for every pairwise comparison the dictionary sort performs (O(k log k) comparisons over k unique day-keys, k ≤ n). Net effect is closer to O(n·k·log k) than a clean O(n²), but it is asymptotically far worse than the O(n) a single `Calendar.startOfDay` grouping pass would cost, and every `rows.first` re-evaluates the *also-recomputed* `rows` property from scratch each time (filteredTransactions/filteredTransfers/upcomingRecurring re-run their own filters), compounding the cost further. The audit's O(n²) characterization is a reasonable simplification of a genuinely worse-than-necessary hot path; confirmed still live, not resolved by any intervening task.
  - `EggSpend/Views/Transactions/TransactionsListView.swift:414-420` — the `DateFormatter.sectionHeader` extension is unchanged and still exactly matches:
    ```swift
    extension DateFormatter {
        static let sectionHeader: DateFormatter = {
            let f = DateFormatter()
            f.dateFormat = "MMMM d, yyyy"
            return f
        }()
    }
    ```
    Fixed `dateFormat` string with no `.locale` assignment — confirmed non-localized (will always render US-style "July 10, 2026" month/day ordering and English month names regardless of device language/region).

A yearly-cadence cluster's oldest-to-newest span can therefore legitimately be up to `2 × 369 = 738` days ≈ 24.3 months — a 24-month (≈730-day) window would clip the oldest of 3 yearly occurrences right at the boundary in the worst case, silently dropping a real yearly subscription down to 2 occurrences (below the `>= 3` floor) and defeating detection entirely — the opposite of R10's goal. 36 months (≈1095 days) leaves roughly 357 days (~1 year) of slack beyond the worst-case yearly span, comfortably covering calendar-month/leap-year arithmetic edge cases in whatever windowing helper T16b adds (e.g. `calendar.date(byAdding: .month, value: -36, to: now)`), while still meaningfully bounding how far back a single anomalous historical charge (like a stale trial price) can reach to poison the `allSatisfy` stability gate for monthly/weekly/biweekly clusters, where 36 months is already generously more history than needed to establish a stable mode amount.
    - Concretely: `SubscriptionAuditView` (or a small helper it calls) should compute `let windowStart = calendar.date(byAdding: .month, value: -36, to: now) ?? .distantPast` and pass `transactions.filter { $0.date >= windowStart }` into `SubscriptionDetector.detect(transactions:recurring:now:calendar:)` — `recurring` (for the active-exclusion set) should NOT be windowed, since an old-but-still-active `RecurringTransaction` must still suppress detection of its title regardless of when it was created.
